# lyra-design-system v3.0.0 — Design Spec

**Date:** 2026-04-15 **Status:** Draft — awaiting review **Related work:** `skills/lyra-design-system/SKILL.tmpl.md`

## Summary

Rewrite `skills/lyra-design-system/SKILL.tmpl.md` to mimic the structure and posture of gstack's `design-consultation` skill
(<https://github.com/garrytan/gstack/blob/main/design-consultation/SKILL.md.tmpl>) while preserving every distinctive piece
of the current Lyra skill. Bump to `version: 3.0.0`.

The rewrite adopts gstack's phased flow (Phase 0–6), its opinionated "consultant not form wizard" posture, its SAFE/RISK
framing, its design-knowledge reference blocks, its coherence-validation rules, and its full `DESIGN.md` schema — then slots
Lyra's existing Reference URL Analysis, Adversarial Review subagent, `moodboard.html` artifact spec, and preflight include
into that shape. During Phase 5, the skill generates **both** Lyra's existing `.design/moodboard.html` and a new gstack-style
`.design/preview.html` so the operator can compare artifacts side-by-side.

## Goals

1. Reshape `lyra-design-system` into gstack's phased structure and consultant posture without losing any capability the
   current skill has.
2. Adopt gstack's concrete design-knowledge reference material (aesthetic directions, decoration levels, layout/color/motion
   approaches, font recs by role, font blacklist, overused-font list, AI-slop anti-patterns, safe/risk framing, coherence
   validation) as operational content inside the skill.
3. Produce both preview artifacts (`moodboard.html` and `preview.html`) in Phase 5 so the operator can directly compare
   Lyra's current visual deliverable against gstack's preview format.
4. Adopt gstack's full `DESIGN.md` schema as the canonical format Lyra writes, extended with Lyra's existing "Signature Move"
   and "Anti-Persona Guardrails" sections.
5. Make the design system load-bearing in future sessions by appending a Design System block to the project's `CLAUDE.md`
   (matching gstack's behavior), pointing at `.design/DESIGN.md`.

## Non-goals

- **Gstack Path A (AI mockup generation via `$D`).** No Lyra equivalent binary, no port.
- **Gstack preamble system.** No `{{PREAMBLE}}`, `{{SLUG_EVAL}}`, `{{BROWSE_SETUP}}`, `{{DESIGN_SETUP}}`,
  `{{LEARNINGS_SEARCH}}`, `{{LEARNINGS_LOG}}`, `{{DESIGN_OUTSIDE_VOICES}}`, `{{DESIGN_SHOTGUN_LOOP}}`, office-hours
  integration, `~/.gstack/` paths, or preamble tiers.
- **`allowed-tools` frontmatter.** Lyra skills don't declare allowed tools in frontmatter.
- **New `lib/` modules.** This rewrite is confined to `SKILL.tmpl.md`. It does not extract anything into `lib/` and does not
  change other skills.
- **Changes to other skills.** Sibling skills (`lyra-typography-system`, `lyra-color-theory`, `lyra-qa-design`, etc.) are
  untouched.
- **Migration of existing `.design/DESIGN.md` files.** The v2.0.0 format is a strict subset of the new schema; old docs keep
  working, but this spec does not define an auto-migration.

### Top-of-file posture block

Immediately after `{{lib/preflight/preflight.md}}` and the `# lyra-design-system` heading, the skill opens with gstack's
consultant posture, adapted to Lyra's voice:

> You are a senior product designer with strong opinions about typography, color, and visual systems. You don't present menus
> — you listen, think, research, and propose. You're opinionated but not dogmatic. You explain your reasoning and welcome
> pushback.
>
> **Your posture:** Design consultant, not form wizard. You propose a complete coherent system, explain why it works, and

### Phase map

The phase numbering matches gstack for direct comparability.

#### Phase 0 — Pre-checks

Merges v2.0.0 Section 0 with gstack Phase 0.

- Inspect `README.md` (top ~60 lines), `package.json` (top ~20 lines), and `src/`/`app/` folders via `Read` / `Glob`.
- Check for existing design doc at `.design/DESIGN.md` **or** `DESIGN.md` **or** `design-system.md` (Lyra path plus gstack
  paths, for forward-compat).
- If an existing doc is found, use gstack's prompt phrasing: _"You already have a design system. Want to **update** it,
  **start fresh**, or **cancel**?"_ — delivered via `AskUserQuestion`.
- If codebase is empty and product purpose is unclear, say so and offer to stop: _"I don't have a clear picture of what
  you're building yet. Want to describe it first, or should I proceed from what I can see?"_

Drops gstack's office-hours file lookup entirely.

#### Phase 0.5 — Reference URL Analysis (Lyra keepsake)

Verbatim from v2.0.0 Section 0.5. If the user provides a reference URL:

1. Fetch and analyse it BEFORE Phase 1 questions.
2. Output a Layout Blueprint extract with the fields: Navigation, Tab bar, Photo chrome, H1 element, Section order, Feature
   row, CTAs, Utility, Colours.
3. Treat the Blueprint as explicit layout truth in Phase 3 (Proposal).

Positioned before Phase 1 so the reference informs discovery questions. Uses `WebFetch`.

#### Phase 1 — Product Context

Replaces v2.0.0's open-ended 4–6 exchange discovery with gstack's single consolidated question.

- One `AskUserQuestion` covering: confirm what the product is / who it's for / space or industry, project type (web app /
  dashboard / marketing site / editorial / internal tool), and whether the user wants competitive research.
- If Phase 0 gathered enough context from README/package.json, pre-fill the confirmation: _"From what I can see, this is [X]
  for [Y] in the [Z] space. Sound right? And would you like me to research what's out there in this space, or should I work
  from what I know?"_
- Explicitly say: _"At any point you can drop into chat and we'll talk through anything — this isn't a rigid form, it's a
  conversation."_

#### Phase 2 — Research (opt-in)

Replaces v2.0.0 Section 1.5 with gstack's three-layer synthesis; keeps Lyra's Visual Gap insight format as the Layer 3
output.

**Step 1 — Identify what's out there via `WebSearch`**: 5–10 products in their space, searching phrases like
`"[category] website design"`, `"best [category] web apps 2025"`.

**Step 2 — Visual research via Playwright MCP (if available)**: Lyra doesn't ship a `browse` binary, but Playwright MCP tools
(`mcp__plugin_playwright_playwright__browser_navigate`, `browser_take_screenshot`, `browser_snapshot`) are available in this
environment. Visit the top 3–5 sites, capture screenshots, collect `browser_snapshot` output for structural data. Gracefully
skip sites that block headless browsers or require login. If Playwright MCP is unavailable, rely on `WebSearch` results plus
built-in design knowledge.

**Step 3 — Three-layer synthesis**:

- **Layer 1 (tried and true):** shared patterns across the category — table stakes.
- **Layer 2 (new and popular):** what current discourse and top results are trending toward.
- **Layer 3 (first principles):** given _this_ product's users and positioning, where should we deliberately break from
  category norms?

**Eureka check:** if Layer 3 reveals a genuine insight, name it explicitly: _"EUREKA: Every [category] product does X because
they assume [assumption]. But this product's users [evidence] — so we should do Y instead."_

**Visual Gap output** (Lyra's existing insight format, kept verbatim, used as the Layer 3 summary):

> "The industry currently uses [Primitive] which communicates [Vibe]. However, this leads to an experience that feels
> [Negative Trait]. By pivoting to [Proposed Primitive], we bridge the gap between [Function] and [Desired Emotion]."

**Graceful degradation**: Playwright MCP + WebSearch → WebSearch only → built-in design knowledge.

If the user declined research in Phase 1, skip Phase 2 entirely.

#### Phase 2.5 — Adversarial Review (Lyra keepsake)

Lyra's v2.0.0 Section 1.75, repositioned after Phase 2 so the subagent can react to research findings.

Dispatch a subagent (via the `Agent` tool, `subagent_type: general-purpose`) with a prompt that includes the Phase 1 product
context and (if present) the Phase 2 Visual Gap insight:

> "Propose an unexpected design direction for this brief. Name specific fonts, hex values, layout approach, and 2 deliberate
> departures from category norms. Be bold."

Use the subagent's response to strengthen the Phase 3 proposal or carry it forward as a "wild alternative" option in Phase 3.

#### Phase 3 — The Complete Proposal

Adopts gstack's full proposal format. Replaces v2.0.0 Section 2.

Single `AskUserQuestion` presenting the complete system with SAFE / RISK breakdown:

```
Based on [product context] and [research findings / my design knowledge]:

AESTHETIC: [direction] — [one-line rationale]
DECORATION: [level] — [why this pairs with the aesthetic]
LAYOUT: [approach] — [why this fits the product type]
COLOR: [approach] + proposed palette (hex values) — [rationale]
TYPOGRAPHY: [3 font recommendations with roles] — [why these fonts]
SPACING: [base unit + density] — [rationale]
MOTION: [approach] — [rationale]

This system is coherent because [explain how choices reinforce each other].

SAFE CHOICES (category baseline — your users expect these):
  - [2-3 decisions that match category conventions, with rationale]

RISKS (where your product gets its own face):
  - [2-3 deliberate departures from convention]
  - For each risk: what it is, why it works, what you gain, what it costs

Which risks appeal to you? Want to see different ones? Or adjust anything else?
```

Options presented: A) Looks great — generate previews. B) Adjust [section]. C) Different risks — show wilder options (this is
where the Phase 2.5 adversarial proposal gets surfaced if not already used). D) Start over with a different direction. E)
Skip previews, write `DESIGN.md` directly.

**Design-knowledge reference blocks** (embedded in the skill as operational content the agent uses to inform its proposal —
**not** displayed to the user as menus):

- **Aesthetic directions** (10 options): Brutally Minimal, Maximalist Chaos, Retro-Futuristic, Luxury/Refined,
  Playful/Toy-like, Editorial/Magazine, Brutalist/Raw, Art Deco, Organic/Natural, Industrial/Utilitarian — each with a
  one-line description.
- **Decoration levels**: minimal / intentional / expressive.
- **Layout approaches**: grid-disciplined / creative-editorial / hybrid.
- **Color approaches**: restrained / balanced / expressive.
- **Motion approaches**: minimal-functional / intentional / expressive.
- **Font recommendations by purpose**: Display/Hero, Body, Data/Tables, Code — each with 4–6 specific recommended families.
- **Font blacklist** (never recommend): Papyrus, Comic Sans, Lobster, Impact, Jokerman, Bleeding Cowboys, Permanent Marker,
  Bradley Hand, Brush Script, Hobo, Trajan, Raleway, Clash Display, Courier New (for body).
- **Overused fonts** (never recommend as primary — comply only if user specifically asks): Inter, Roboto, Arial, Helvetica,
  Open Sans, Lato, Montserrat, Poppins. This list merges gstack's list with Lyra's current "never use" list (which flagged
  the same fonts).
- **AI slop anti-patterns** (never include in recommendations): purple/violet gradient accents, 3-column feature grid with
  icons in colored circles, uniform centered spacing, uniform bubbly border-radius, gradient primary buttons, generic
  stock-hero sections, "Built for X / Designed for Y" marketing copy.

**Coherence Validation**: when the user overrides one section, check if the rest still coheres and flag mismatches with a
gentle nudge — never block. Explicit nudge examples included in the skill text:

- Brutalist/Minimal aesthetic + expressive motion
- Expressive color + restrained decoration
- Creative-editorial layout + data-heavy product

Always accept the user's final choice.

#### Phase 4 — Drill-downs

Adopts gstack Phase 4 verbatim. Triggered when the user picks option B in Phase 3.

- **Fonts**: 3–5 specific candidates with rationale; explain what each evokes; offer to defer the visual preview to Phase 5
  rather than generating a fonts-only preview mid-flow.
- **Colors**: 2–3 palette options with hex values; explain the color theory reasoning.
- **Aesthetic**: walk through which directions fit the product and why.
- **Layout / Spacing / Motion**: present the approaches with concrete tradeoffs for the product type.

Each drill-down is one focused `AskUserQuestion`. After the user decides, re-run the Coherence Validation check from Phase 3
against the updated proposal.

#### Phase 5 — Visual Previews (both artifacts)

Generates **two** self-contained HTML files in `.design/`. Neither uses external frameworks; only Google Fonts is allowed as
an external dependency.

**`.design/moodboard.html`** — Lyra's existing v2.0.0 spec, kept verbatim:

- CSS-only editorial showcase. No JS. No external CDNs except Google Fonts. Opens offline.
- Background must be intentional (off-white or deep dark, never `#FFFFFF`). Bordered elements MUST use padding; add > 1
  spacing block between bordered elements.
- Real contextual content tied to the product — no Lorem Ipsum, no placeholders.
- Display scales 80–140px. Monospace reserved exclusively for actual code or token values.
- Never use Inter, Roboto, Arial, Helvetica, Montserrat, Poppins, Open Sans, or Lato.
- Icons: actual inline SVGs pulled from Lucide, Heroicons, Phosphor, or Tabler — never approximate paths, never emoji.
- Mobile Preview: two 393px-wide mobile frames with realistic layouts (status bar, tab bar, scale, interaction state). Stand
  side-by-side on desktop, stack responsively.

**`.design/preview.html`** — gstack's Phase 5 Path B spec:

- Single self-contained HTML file. Google Fonts via `<link>` tags. Uses the proposed palette throughout (dogfood the system).
  Product name as the hero heading, not Lorem Ipsum.
- **Font specimen section**: each font candidate shown in its proposed role (hero, body, button label, data table row);
  side-by-side comparison if multiple candidates per role; real content matching the product.
- **Color palette section**: swatches with hex values and names; sample UI components in the palette — buttons (primary,
  secondary, ghost), cards, form inputs, alerts (success, warning, error, info); background/text pairings showing contrast.
- **Realistic product mockups**: 2–3 realistic page layouts based on the Phase 1 project type:
  - Dashboard / web app: data table with metrics, sidebar nav, header with user avatar, stat cards.
  - Marketing site: hero with real copy, feature highlights, testimonial, CTA.
  - Settings / admin: form with labeled inputs, toggle switches, dropdowns, save button.
  - Auth / onboarding: login form with social buttons, branding, input validation states.
- **Light/dark mode toggle** using CSS custom properties and a small JS toggle button. (The skill explicitly calls out that
  `preview.html` is allowed JS here; `moodboard.html` is not.)
- **Responsive** on any screen width. **Clean, professional layout** — treat the file as a taste signal.

After both files are written, the skill runs `open .design/moodboard.html .design/preview.html` (macOS — matching the Lyra
target platform) and tells the user:

> "Two previews in `.design/`: `moodboard.html` (Lyra's editorial showcase spec) and `preview.html` (gstack-style preview
> page). Both opened in your browser so you can compare. Tell me which direction feels right, what to adjust, or if you want
> to iterate on either one."

If `open` fails (headless environment), print both paths and tell the user to open them manually.

If the user picked Phase 3 option E (skip previews), skip this phase entirely and go to Phase 6.

**Drops gstack Path A entirely** — no `$D` mockup generation.

### Anti-patterns and Hard Constraints (footer)

Merges gstack's "Important Rules" with Lyra's v2.0.0 "Anti-patterns & Hard Constraints" into one footer section. Deduplicated
— the overlapping items (never recommend blacklisted fonts, never recommend gradients indiscriminately, never approximate SVG
paths) appear once.

Final list:

1. **Propose, don't present menus.** You are a consultant, not a form. Make opinionated recommendations based on the product
   context, then let the user adjust.
2. **Every recommendation needs a rationale.** Never say "I recommend X" without "because Y."
3. **Coherence over individual choices.** A system where every piece reinforces every other piece beats a system with
   individually "optimal" but mismatched choices.
4. **Never recommend blacklisted or overused fonts as primary.** If the user specifically requests one, comply but explain
   the tradeoff.
5. **Both preview files must be beautiful.** They're the first visual output and set the tone for the whole skill.
6. **Conversational tone.** This isn't a rigid workflow. If the user wants to talk through a decision, engage as a thoughtful
   design partner.
7. **Accept the user's final choice.** Nudge on coherence issues, but never block or refuse to write `DESIGN.md` because you
   disagree with a choice.
8. **No AI slop in your own output.** Your recommendations, preview files, and `DESIGN.md` must demonstrate the taste you're
   asking the user to adopt.
9. **NEVER propose "a clean dark theme"** — use explicit hex values.
10. **NEVER use gradients indiscriminately** (especially 2-color linear).
11. **NEVER use generic SVGs without pulling exact paths** from Lucide/Heroicons/Phosphor/Tabler.
12. **Always ensure design decisions follow the emotional brief** (diffuse shadows for soft/premium; raw borders for utility;
    etc.).
